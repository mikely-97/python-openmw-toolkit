# Hub World

A self-contained progression zone for OpenMW built with `python-openmw-toolkit`.

## Zone Overview

```
Hub World  ──── Garden (open)
               ──── Forest (Forest Pass)
               ──── Mine   (Mine Pass)
```

| Zone       | Content                                  | Unlock             |
|------------|------------------------------------------|--------------------|
| Hub World  | Vendor, Bed, Anvil, Forge                | Always open        |
| Garden     | 5 Basil + 5 Mint plants (24 h respawn)   | Free               |
| Forest     | 6 Trees (7-day respawn) + 8 Mushrooms    | Buy Forest Pass    |
| Mine       | 8 Iron + 4 Stone deposits (no auto-respawn)| Buy Mine Pass    |

## Gameplay Loop

1. **Gather** herbs in the Garden → brew Restore Health / Fatigue potions
   at the alchemy apparatus (buy from vendor).
2. **Chop trees** in the Forest with an axe equipped — each click is a swing.
   Axe skill determines damage per swing; Iron Axe is faster.
3. **Craft** at the Anvil: 1 log + 2 branches → Crafted Wooden Axe;
   3 iron ingots → Iron Pickaxe. Success chance scales with Repair skill.
4. **Mine** in the Mine with a pickaxe equipped. Blunt skill determines
   success chance and ore quantity. Deposits deplete permanently until
   you buy a Mine Refresh Token and use the Mine Management Board in the Hub.
5. **Smelt** at the Forge: 2 iron ore → 1 iron ingot (up to 5 batches per use).
6. **Sell** everything to the vendor; buy better tools and passes.

## Quick Setup

### 1. Generate meshes (Blender)

```bash
blender --background --python generate_meshes.py
```

This writes `.dae` files to `data/meshes/hub_world/`.

> **Note — Flatpak OpenMW required on Void Linux** (and likely other distros).
> The system `openmw` package ships an incompatible `osgdb_dae.so`.
> Use: `flatpak run --command=openmw org.openmw.OpenMW`

> **Better Collada Exporter** (optional but recommended):
> Install the Godot/BetterColladaExporter Blender add-on from
> <https://github.com/godotengine/collada-exporter> for cleaner output.

### 2. Build the addon

```bash
cd hub_world/
poetry run python build.py
# → hub_world.omwaddon  (+ validation output)
```

### 3. Configure OpenMW

Append to `~/.var/app/org.openmw.OpenMW/config/openmw/openmw.cfg`:

```ini
data="<absolute-path-to>/hub_world/data"

content=template.omwgame          # base game (must come first)
content=hub_world.omwaddon
content=hub_world.omwscripts
```

The `data=` line exposes the custom meshes to the OpenMW VFS.

> If you are using the example suite, also keep the existing `data=` and
> `content=` lines for `template.omwgame`, `the_hub.omwaddon`, etc.

### 4. Play

Start OpenMW.  The player starts in the Hub World cell (depends on your
`template.omwgame` start cell setup — you may need to `tcl` teleport via
the in-game console: `player->coc "Hub World"`).

## File Structure

```
hub_world/
├── build.py                    ← Run to regenerate hub_world.omwaddon
├── hub_world.omwaddon          ← Generated binary addon
├── hub_world.omwscripts        ← Lua script manifest
├── generate_meshes.py          ← Blender mesh generator
├── data/
│   ├── meshes/hub_world/       ← .dae files (generated by Blender)
│   └── scripts/hub/
│       ├── global.lua          ← All game-state logic (inventory, skills)
│       ├── activator.lua       ← Forwards activation events to global
│       └── player.lua          ← Displays HUD messages
└── README.md
```

## Expanding the Addon

### Add a new harvestable plant

1. In `build.py`: `addon.add_activator("hw_plant_newherb_01", ...)` and place it.
2. In `global.lua` `HARVEST_CFG`: add `hw_plant_newherb_01 = { item="hw_newherb", respawn=… }`.
3. In `activator.lua` `HW_IDS`: add `hw_plant_newherb_01=true`.
4. Add the corresponding `INGR` record in `build.py`.

### Add a new crafting recipe

In `global.lua` `ANVIL_RECIPES`, append a new recipe table entry.

### Add a new zone

1. Add a new `addon.add_interior_cell("Hub NewZone", …)` in `build.py`.
2. Add a new door pair (ACTI in Hub + ACTI in new cell) and wire them in
   `DOOR_DESTINATIONS` in `global.lua`.
3. Optionally add a pass item and check in `DOOR_DESTINATIONS.require`.

## Known Limitations / TODOs

| Item | Status |
|------|--------|
| **Time advance from bed** | Lua API for advancing game time not yet in OpenMW 0.49. Bed currently restores stats only. When `world.advanceTime()` is available, wire it in `handleBed()` in `global.lua`. |
| **Tool durability decay** | The pickaxe/axe condition does not decrease on use. The API (`types.Weapon.condition`) will be wired once the binding is confirmed stable. |
| **Tree/mineral visual state** | Objects don't change appearance when depleted/felled. Full visual states require either separate ACTI records per state or OpenMW's `world.setObjectActive()` (needs careful global-script management). |
| **Training Dummy** | Mentioned in design doc but not yet placed. Add as ACTI with a Blunt/Axe-skill-training handler in `global.lua`. |
| **Mine generation counter** | Refresh clears depletion by incrementing a global generation counter that activators compare against. Full per-instance clearing requires iterating storage keys (API-version dependent). |
| **Custom mesh quality** | `generate_meshes.py` produces basic primitives. Replace `.dae` files with hand-modelled assets as the project matures. |
